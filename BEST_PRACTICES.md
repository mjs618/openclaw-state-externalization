# 最佳实践指南

经过多个项目的实战总结，这些实践能最大化状态外化的价值。

---

## 🎯 1. 任务粒度控制

### 黄金法则: 每个任务 2-4 个 Session 完成

| 粒度 | 说明 | 示例 |
|------|------|------|
| **太大** | 需要 10+ Session | ❌ "重写整个系统" |
| **合适** | 2-4 Session | ✅ "重写用户认证模块" |
| **太小** | 1 Session 内完成 | ⚠️ 没必要外化 |

**拆分技巧**:
```
大任务: 重构电商系统
  ├── 子任务 1: 重构用户模块 (2 Session)
  ├── 子任务 2: 重构商品模块 (3 Session)
  ├── 子任务 3: 重构订单模块 (3 Session)
  └── 子任务 4: 集成测试 (2 Session)
```

## 📝 2. Session 记录规范

### 必须记录的内容

```markdown
## Session XXX: 标题

### ✅ 完成的工作
1. 具体完成项 1
2. 具体完成项 2

### 🐛 遇到的问题
| 问题 | 解决方案 | 状态 |
|------|----------|------|
| 描述 | 如何解决的 | 已解决/待跟进 |

### 💭 关键决策
- **决策 1**: 理由
- **决策 2**: 理由

### 🔄 下一步
- [ ] 待办 1
- [ ] 待办 2
```

### 不要记录的内容

❌ 琐碎的操作日志（"打开了文件 X"）  
❌ 完整的代码（应该在 git 里）  
❌ 临时想法（应该在 `findings.md`）

## 🗂️ 3. 决策管理

### 何时创建决策记录？

当满足以下条件之一：
- 选择了某个技术方案（A vs B）
- 改变了原有设计
- 解决了争议性问题
- 影响了项目方向

### 决策记录模板

```markdown
# 决策 005: 选择数据库

## 背景
需要在 PostgreSQL 和 MongoDB 之间选择

## 选项对比
| 维度 | PostgreSQL | MongoDB |
|------|------------|---------|
| 事务支持 | ✅ | ⚠️ |
| 查询复杂度 | 强 | 中等 |
| 团队熟悉度 | 高 | 低 |

## 决定
选择 PostgreSQL

## 理由
1. 需要复杂查询和事务支持
2. 团队更熟悉
3. 已有运维经验

## 影响
- 需要设计关系型 schema
- 考虑读写分离架构

## 时间
2026-03-06
```

## ⏰ 5. 时间规划

### Session 时长建议

| 类型 | 时长 | 适合场景 |
|------|------|----------|
| **快速 Session** | 30-60 min | 简单任务、修复 bug |
| **标准 Session** | 2-4 hours | 常规开发工作 |
| **深度 Session** | 4-8 hours | 复杂设计、架构决策 |

### 每日节奏

```
09:00 启动 Session
  └─ 读取状态 (5min)
  └─ 确认今日目标 (5min)
  └─ 开始工作

12:00 午餐休息
  └─ 结束 Session，保存状态

14:00 启动新 Session
  └─ 恢复上午状态
  └─ 继续工作

18:00 结束工作
  └─ 结束 Session，保存状态
  └─ git commit & push
```

## 🔄 6. Git 工作流

### 推荐分支策略

```bash
# main 分支: 稳定状态
git checkout main
git pull

# 大改动用 feature 分支
git checkout -b feature/new-auth
# ... 工作 ...
git add state/
git commit -m "Session 005: 新认证模块"
git push origin feature/new-auth
# PR 合并到 main
```

### Commit 消息规范

```
Session XXX: 一句话总结

- 完成的工作 1
- 完成的工作 2
- 关键决策
```

示例:
```
Session 003: 完成数据库设计

- 设计了 User/Product/Order 三张表
- 确定了外键关系
- 决策: 使用软删除而非硬删除
```

## 🚨 8. 常见陷阱

### ❌ 陷阱 1: 过度记录

**症状**: 每个小操作都记录，文件膨胀

**解决**:
- 只记录"为什么"和"结果"
- 不记录"怎么做"（在 git 里）

### ❌ 陷阱 2: 状态不同步

**症状**: `manifest.json` 和实际状态不一致

**解决**:
- Session 结束时必须更新 manifest
- 使用脚本自动同步

### ❌ 陷阱 3: 长时间不 commit

**症状**: 一天后才 push，冲突难以解决

**解决**:
- 每个 Session 结束都 commit
- 设置 auto-commit 钩子

### ❌ 陷阱 4: 任务边界不清

**症状**: 多个任务混在一起，状态混乱

**解决**:
- 使用 `task-tree.md` 明确拆分
- 一个 Session 聚焦一个子任务

## 🚀 10. 持续改进

### 定期回顾

每完成一个项目：
1. 回顾 `state/sessions/` 记录
2. 总结哪些状态有用，哪些没用
3. 优化模板
4. 更新最佳实践

### 分享经验

- 提交 PR 改进模板

- 帮助其他用户

## 总结

好的状态外化 = **恰到好处**的记录 + **清晰**的结构 + **及时**的同步

不是越多越好，而是**在需要的时候能找到需要的信息**。
